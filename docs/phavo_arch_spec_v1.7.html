Phavo — Technical Architecture Specification

Version 1.0 PRD ref v1.8 Phase 1 · Web Dashboard Stack SvelteKit · Hono · libSQL · Better Auth · Bun

System Architecture

Phavo Phase 1 is a single-container application. The SvelteKit server handles both SSR and the Hono API. @phavo/agent runs as an in-process library (not a daemon) and is called directly by Hono route handlers. All state is stored in a local libSQL file on the Docker volume.

Encryption Model

All sensitive values (credentials, widget configs containing secrets, TOTP secrets) are encrypted using AES-256-GCM via the Web Crypto API (crypto.subtle), available natively in Bun.

Ciphertext Format

All encrypted values stored as a single base64 string with a 3-part layout:

base64url( iv[12 bytes] + authTag[16 bytes] + ciphertext[n bytes] )

async function encrypt(plaintext: string): Promise<string> {
  const iv = crypto.getRandomValues(new Uint8Array(12))
  const enc = await crypto.subtle.encrypt(
    { name: 'AES-GCM', iv, tagLength: 128 },
    derivedKey,
    new TextEncoder().encode(plaintext)
  )
  // authTag is appended to ciphertext by SubtleCrypto automatically
  const combined = new Uint8Array(iv.length + enc.byteLength)
  combined.set(iv, 0)
  combined.set(new Uint8Array(enc), iv.length)
  return btoa(String.fromCharCode(...combined))
}

Auth Flow — Free / Standard (phavo.io)

Free and Standard users authenticate via phavo.io OAuth. The local Phavo server never sees or stores phavo.io passwords.

Browser
Phavo Server
phavo.io
libSQL
1. Browser    ──── GET /auth/login ──────────────────────►  Server
2.            ◄─── redirect → phavo.io/oauth/authorize ──  Server
3. Browser    ──── user logs in ────────────────────────►  phavo.io
4.            ◄─── redirect /auth/callback?code=xxx ─────  phavo.io
5. Browser    ──── GET /auth/callback?code=xxx ──────────►  Server
6.            Server  ─── POST /api/oauth/token ──────────►  phavo.io
7.                    ◄── { access_token, user } ──────────  phavo.io
8.            Server  ─── POST /api/license/validate ──────►  phavo.io
9.                    ◄── { tier: 'free'|'standard' } ──────  phavo.io
10.           Server  ─── INSERT sessions({ tier, graceUntil }) ─►  libSQL
11.           Server  ─── INSERT/UPDATE users ───────────────►  libSQL
12. Browser   ◄─── Set-Cookie: session=<32-byte-token>  ──  Server
                 (HttpOnly, Secure, SameSite=Strict)
13. Browser   ◄─── redirect → / (dashboard) ──────────────  Server
Grace period: At step 9, the server calculates graceUntil = Date.now() + (tier === 'free' ? 86400000 : 259200000). If phavo.io is unreachable at step 8, the server checks whether the existing session's graceUntil is still in the future. If yes, the existing session is refreshed. If no, login is rejected with a "phavo.io unreachable — grace period expired" error.

Auth Flow — Local Tier

Local tier has two sub-flows: first activation (online, once) and subsequent logins (fully offline).

A. First Activation

1. Browser  ──── POST /api/v1/auth/login { licenseKey, username, password } ─►  Server
2.          Server  ─── POST /api/license/activate { licenseKey, instanceId } ──►  phavo.io
3.                  ◄── { activationJwt: <RS256-signed-JWT> } ─────────────────  phavo.io
4.          Server  ─── INSERT licenseActivation({ activationJwt }) ─────────────►  libSQL
5.          Server  ─── INSERT users({ passwordHash: argon2id(password) }) ────►  libSQL
6.          Server  ─── INSERT sessions({ tier: 'local', graceUntil: null }) ──►  libSQL
7. Browser  ◄──────── Set-Cookie: session=<32-byte-token> ──────────────────  Server

B. Subsequent Logins (fully offline)

1. Browser  ──── POST /api/v1/auth/login { username, password } ─────────────►  Server
2.          Server  ─── SELECT passwordHash FROM users ────────────────────────►  libSQL
3.          Server  ─── argon2id.verify(password, hash) [in-process]
4.          Server  ─── SELECT activationJwt FROM licenseActivation ───────────►  libSQL
5.          Server  ─── verifyRS256(jwt, EMBEDDED_PUBLIC_KEY) [in-process]
6.          Server  ─── INSERT sessions({ tier: 'local' }) ─────────────────────►  libSQL
7. Browser  ◄──────── Set-Cookie: session=<32-byte-token> ──────────────────  Server
                   ← No network call to phavo.io required at any point

Session Validation (per-request)

Runs on every request via authMiddleware in the Hono chain, before any route handler executes.

1. Request  ──── Cookie: session=<token> ───────────────────────────────────►  authMiddleware
2.          ─── SELECT * FROM sessions WHERE id = token ─────────────────────►  libSQL
            ↳ not found? → 401 Unauthorized
            ↳ expiresAt < now? → 401, delete session
            ↳ authMode = 'phavo-io' AND graceUntil < now? → 401 "grace period expired"
3.          ─── c.set('session', sessionRecord) [stored in Hono context]
4.          ─── call next() → requireTier() → route handler

Tier Enforcement

Middleware Implementation

// apps/web/src/lib/server/middleware/requireTier.ts

const TIER_RANK: Record<Tier, number> = { free: 0, standard: 1, local: 2 }

export function requireTier(minimum: Tier) {
  return async (c: Context, next: Next) => {
    const session = c.get('session')         // set by authMiddleware, already DB-validated
    if (!session) return c.json(err('Unauthorized'), 401)

    if (TIER_RANK[session.tier] < TIER_RANK[minimum]) {
      return c.json(err('Upgrade required'), 403)
    }
    await next()
  }
}

export function requireSession() {
  return async (c: Context, next: Next) => {
    if (!c.get('session')) return c.json(err('Unauthorized'), 401)
    await next()
  }
}

Widget Manifest — Split Design

The widget manifest endpoint returns two kinds of entries in a single array. This resolves the tension between security (never send Standard secrets to Free clients) and conversion UX (Free users need to see locked widgets to be prompted to upgrade).

FieldWidgetDefinition (full)WidgetTeaserDefinition (locked)

name✓✓
description✓✓

locked— (absent)true
dataEndpoint✓✗ never sent
configSchema✓✗ never sent

The frontend Widget Drawer renders a locked card for each teaser entry — name, description, and a lock icon. Clicking triggers the upgrade modal. The client has zero information about how to call the locked widget's data endpoint, making client-side bypass structurally impossible.

Widget Data Flow

Widget data is always polled client-side at the refreshInterval defined in the widget's manifest. No WebSocket, no SSE. The dashboard keeps a polling map keyed by widget instance ID.

// apps/web/src/lib/stores/widgets.svelte.ts
const pollingMap = new Map<string, ReturnType<typeof setInterval>>()

function startPolling(instance: WidgetInstance, def: WidgetDefinition) {
  const fetch = async () => {
    const res = await fetch(def.dataEndpoint)
    const json: ApiResponse<unknown> = await res.json()
    widgetData[instance.id] = json.ok ? { data: json.data, error: null } : { data: null, error: json.error }
  }
  fetch()  // immediate first fetch
  pollingMap.set(instance.id, setInterval(fetch, def.refreshInterval))
}

function stopPolling(instanceId: string) {
  clearInterval(pollingMap.get(instanceId))
  pollingMap.delete(instanceId)
}

Each <WidgetCard> component subscribes to widgetData[instance.id]. While data is undefined, it renders <Skeleton>. On error, it renders the error state with a retry button. Errors never crash the dashboard — each widget is independent.

  | 'network:outbound'    // external HTTP requests (internet)
  | 'network:local'       // LAN hosts (Pi-hole, local APIs)
  | 'db:read'             // read from phavo DB via WidgetContext.db
  | 'db:write'            // write to phavo DB via WidgetContext.db
  | 'credentials:read'    // decrypt credentials via WidgetContext.credentials
  | 'fs:read'             // read /data/plugins/<own-id>/ only
  | 'notify'              // push notifications via WidgetContext.notify()
  | 'agent:metrics'       // call @phavo/agent metric functions

Permissions declared in manifest.json are shown to the user before the plugin activates. At runtime, the permission middleware blocks any call to a capability not in the manifest — it does not rely on the plugin being "well-behaved".

Sandboxing caveat (Phase 1.x): Plugins run in the same Bun process as the SvelteKit server. The import allowlist and permission middleware are code-level guards, not OS-level isolation. A sophisticated plugin could theoretically bypass these using Bun's native APIs. This is an acceptable trade-off for a self-hosted tool where the user chose to install the plugin. Phase 4 marketplace plugins use process-level sandboxing.

// handler.ts
import { defineWidget, createHandler, PluginError, z } from '@phavo/plugin-sdk'

export const manifest = defineWidget({
  id:              'com.example.my-widget',
  name:            'My Widget',
  description:     'Fetches data from example.com',
  version:         '1.0.0',
  author:          'your-phavo-io-username',
  schemaVersion:   '1',
  category:        'integration',
  tier:            'standard',
  permissions:     ['network:outbound', 'credentials:read'],
  defaultSize:     { w: 6, h: 4 },
  minSize:         { w: 3, h: 2 },
  sizes:           ['S', 'M', 'L'],
  refreshInterval: 60_000,
  configSchema: z.object({
    apiUrl: z.string().url().meta({ label: 'API URL' }),
    apiKey: z.string().min(1).meta({ label: 'API Key', credential: true }),
  }),
})

export default createHandler(async (ctx) => {
  const apiKey = await ctx.credentials.get('apiKey')
  if (!apiKey) throw new PluginError('API key not configured', 400)
  const res = await fetch(`https://api.example.com/data?key=${apiKey}`)
  if (!res.ok) throw new PluginError('External API unreachable', 503)
  return res.json()
  // createHandler wraps response as { ok: true, data: T }
  // PluginError → { ok: false, error: message } with correct status code
})

Platform Abstraction

All server code must read paths and ports from environment variables. This is the single architectural prerequisite that allows the same apps/web/ codebase to run identically in Docker and as a Tauri desktop sidecar — without any conditional logic or platform-specific branches in the application code itself.

installMethod Derivation

On first start, installMethod is derived from env.platform and written to the config table. It is never re-derived — subsequent starts read from DB.

// apps/web/src/lib/server/install.ts
export async function deriveInstallMethod(): Promise<InstallMethod> {
  const existing = await db.getConfig('installMethod')
  if (existing) return existing as InstallMethod

  const method: InstallMethod =
    env.platform === 'tauri'  ? 'tauri'       :
    env.platform === 'bun'    ? 'bun-direct'  :
    isDockerCompose()         ? 'docker-compose' : 'docker-run'

  await db.setConfig('installMethod', method)
  return method
}

// isDockerCompose(): checks if /.dockerenv exists AND docker-compose label is present
// Fallback: 'docker-run' for plain docker run invocations

Widget States

Every widget instance on the dashboard is in exactly one of five states at all times. The state is determined client-side from the combination of polling result and config presence.

type WidgetState =
  | 'loading'        // initial fetch in progress — show skeleton
  | 'active'         // last fetch succeeded — show data
  | 'unconfigured'   // configSchema exists but no saved config — show configure prompt
  | 'error'          // fetch failed — show error + retry
  | 'stale'          // previous fetch succeeded, current failing — show stale data + warning

State Transitions

Widget added to dashboard
      │
      ▼
check configSchema
  ├─ no configSchema → loading → startPolling()
  └─ has configSchema → check credentials in DB
       ├─ credentials exist → loading → startPolling()
       └─ no credentials   → unconfigured (no polling started)

loading
  ├─ fetch success  → active
  └─ fetch error    → error

active
  ├─ next fetch success → stay active
  └─ next fetch error   → stale (keep last data, show warning)

stale
  ├─ next fetch success → active
  └─ after 3 failures   → error (clear stale data)

error
  └─ user clicks retry  → loading

unconfigured
  └─ user saves config  → loading → startPolling()

SchemaRenderer — Auto-Generated Settings UI

Every widget with a configSchema automatically gets a Settings panel. The <SchemaRenderer> component reads the Zod schema from the widget manifest and renders appropriate form fields. No per-widget custom Settings code is needed.

Iron rule: Never write a hardcoded settings panel for a specific widget. If Pi-hole needs a URL + token, that is expressed in its configSchema, not in a custom Svelte component. <SchemaRenderer> handles all cases.

Schema Version Compatibility

On widget instance load (after server restart or widget update):

const savedSchemaVersion = instance.configSchemaVersion  // widgetInstances.configSchemaVersion — the widget version when config was saved
const currentSchemaVersion = def.version  // WidgetDefinition.version — current version from registry
// Note: configSchemaVersion tracks the WIDGET version for config migration.
// The separate schemaVersion in plugin manifests tracks PLUGIN SDK API compatibility — distinct concept.

if (isBreakingChange(savedSchemaVersion, currentSchemaVersion)) {
  // Existing config may be incompatible
  instance.state = 'unconfigured'
  notify({ type: 'widget-warning',
    title: `${def.name} needs reconfiguration`,
    body: 'Widget was updated and requires new settings.' })
} else {
  // Additive change: merge new optional fields with defaults
  instance.config = { ...defaults, ...instance.config }
}
Breaking change definition: Any change that removes or renames a required field in configSchema. Adding new optional fields is always safe. Widget authors must bump the semver minor version on any breaking schema change.

Update Mechanism

Backend — GitHub Releases Polling

// apps/web/src/lib/server/update.ts

const UPDATE_CACHE_TTL = 3_600_000  // 1 hour in ms
let cachedResult: UpdateCheckResult | null = null
let cachedAt = 0

export async function checkForUpdate(): Promise<UpdateCheckResult> {
  if (cachedResult && Date.now() - cachedAt < UPDATE_CACHE_TTL) return cachedResult

  const res = await fetch('https://api.github.com/repos/phabioo/phavo/releases/latest', {
    headers: { 'User-Agent': 'phavo-self-hosted' }
  })
  if (!res.ok) return { updateAvailable: false, current: PHAVO_VERSION, latest: PHAVO_VERSION }

  const release = await res.json()
  cachedResult = {
    current:         PHAVO_VERSION,
    latest:          release.tag_name.replace('v', ''),
    updateAvailable: semverGt(release.tag_name, PHAVO_VERSION),
    changelog:       release.body ?? null   // raw Markdown from release notes
  }
  cachedAt = Date.now()
  return cachedResult
}

interface UpdateCheckResult {
  current:         string
  latest:          string
  updateAvailable: boolean
  changelog:       string | null   // Markdown, rendered in frontend
}
PHAVO_VERSION is injected at build time via Bun.env.npm_package_version from package.json. It is never configurable at runtime.

// Zod schema (server-side validation via POST /api/v1/widgets/:instanceId/config)
const PiholeConfigSchema = z.object({
  url:   z.string().url(),              // e.g. "http://pi.hole" or "http://192.168.1.10"

Widget Config Endpoint

POST /api/v1/widgets/:instanceId/config any session (tier check per widget) Save config for a specific widget instance
Request
{ config: unknown }
// Server: looks up widgetId from instanceId → finds WidgetDefinition.configSchema
// Validates request.config against configSchema (Zod) server-side
// Splits config: credentials → credentials table (encrypted), rest → widgetInstances.configEncrypted
// Returns 400 if Zod validation fails
Response 200
{ ok: true, data: { saved: true } }
Credential separation rule: The configSchema for each widget annotates which fields are credentials using a Zod .meta({ credential: true }) marker. The save handler iterates schema fields, writes credential: true fields to credentials table (encrypted, keyed as widgetId_fieldName), and writes the rest to widgetInstances.configEncrypted. Credentials are never included in config exports unless explicitly requested with a passphrase.

Import / Export — Full Spec

Config Import/Export is fully specified in the API sections (Config Endpoints). This section covers the frontend behaviour and edge cases.

Export Flow

async function handleExport(includeCredentials: boolean, passphrase?: string) {
  const url = new URL('/api/v1/config/export', window.location.origin)
  if (includeCredentials) url.searchParams.set('includeCredentials', 'true')

  const res = await fetch(url.toString(), {
    method: includeCredentials ? 'POST' : 'GET',
    body:   includeCredentials ? JSON.stringify({ passphrase }) : undefined
  })
  const blob = await res.blob()
  const a = document.createElement('a')
  a.href = URL.createObjectURL(blob)
  a.download = `phavo-config-${new Date().toISOString().slice(0,10)}.json`
  a.click()
}

Content Security Policy

Applied as an HTTP response header on all dashboard pages. Managed as a Hono middleware.

Content-Security-Policy:
  default-src 'self';
  script-src  'self';                            /* no inline scripts */
  style-src   'self' https://fonts.googleapis.com;
  font-src    'self' https://fonts.gstatic.com;
  img-src     'self' data: https:;               /* data: for widget icons */

Plugin Svelte components cannot load external resources unless the user explicitly adds exceptions via Settings → Security. The default CSP blocks all external script and connect-src for plugin content.
