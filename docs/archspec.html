Version 3.2 Date 2026-04-10 PRD v4.1 Roadmap v5.0 Runtime v0.8.1 Design Celestial Wish

Overview

Phavo is a self-hosted personal dashboard. It runs entirely locally — no cloud account, no subscription, no phone-home. Two tiers: Stellar (free) and Celestial (one-time purchase, Ed25519 offline license).

Key architectural invariant: The app remains local-first. Runtime outbound calls are limited to explicit integrations (update check, weather API, RSS feeds, Pi-hole). Auth/licensing is fully local-only.

The web app (apps/web/) is the only active runtime target. Desktop (apps/desktop/) and mobile stubs exist for future Tauri integration.

Middleware Stack

Every authenticated, non-GET route passes through this stack in order:

1authMiddlewareReads session cookie → SELECT sessions WHERE id = token AND expiresAt > now(). Injects ctx.session. Returns 401 if missing.
2csrfMiddlewareDouble-submit HMAC pattern. All non-GET mutations require X-CSRF-Token header derived from session ID. Returns 403 on mismatch.
3rateLimitMiddlewarePer-IP (unauthenticated) or per-session (authenticated) counters. In-memory (Phase 1). Endpoint-specific limits. Returns 429 on breach.
4requireTier(tier)Applied per-route. Re-reads tier from licenseActivation table — never from cookie. Returns 403 if session.tier is below required tier.

CSRF — fetchWithCsrf()

// Client-side helper — all mutating store methods use this
async function fetchWithCsrf(url: string, options: RequestInit) {
  const token = deriveToken(sessionId)  // HMAC-SHA256(sessionId, secret)
  return fetch(url, {
    ...options,
    headers: { ...(options.headers ?? {}), 'X-CSRF-Token': token }
  })
}

Auth & Session

Local-only auth. Runtime uses local username/password (Argon2id), local session enforcement, and offline Ed25519 license verification.

Login flow

POST /api/v1/auth/login  { username, password }

1. SELECT user WHERE username = ?
2. argon2id.verify(storedHash, password)  → 401 on fail
3. INSERT sessions { id: randomBytes(32), userId, tier, expiresAt: +7days }
4. Set-Cookie: phavo_session=<sessionId>; HttpOnly; SameSite=Strict; Secure
5. Return { ok: true, data: { loggedIn: true } }
   // tier is NOT returned to client — never serialised into HTML or stores

Session record

sessions.tier is the single source of truth. It is derived from the licenseActivation table on login. Cookie modification has no effect — tier is re-read from DB on every requireTier() call.

TOTP (optional 2FA)

POST /api/v1/auth/totp/setup   → returns QR code + backup codes (AES-encrypted at rest)
POST /api/v1/auth/totp/verify  { token }  → validates TOTP before completing login

Tier Enforcement

requireTier() — how it works

function requireTier(minTier: 'stellar' | 'celestial') {
  return async (c: Context, next: Next) => {
    const session = c.get('session')           // injected by authMiddleware
    const rank = { stellar: 0, celestial: 1 }
    if (rank[session.tier] < rank[minTier])
      return c.json({ ok: false, error: 'Celestial tier required' }, 403)
    await next()
  }
}

Encryption Model

Algorithm scope: AES-256-GCM = data at rest (credentials, config exports). Ed25519 = digital signatures only (license key verification, plugin signing, Tauri updates). Argon2id = password hashing (one-way). PBKDF2-SHA256 = passphrase-protected config export key derivation only.

Encrypt / Decrypt

async function encrypt(plaintext: string): Promise<string> {
  const iv  = crypto.getRandomValues(new Uint8Array(12))
  const enc = await crypto.subtle.encrypt({ name: 'AES-GCM', iv }, aesKey, encode(plaintext))
  return btoa([ ...iv, ...new Uint8Array(enc) ].join(','))  // iv(12) + ciphertext+authTag
}

async function decrypt(stored: string): Promise<string> {
  const bytes = new Uint8Array(atob(stored).split(',').map(Number))
  const iv  = bytes.slice(0, 12)
  const ct  = bytes.slice(12)
  const dec = await crypto.subtle.decrypt({ name: 'AES-GCM', iv }, aesKey, ct)
  return decode(dec)
}

+layout.server.ts — per-request

export const load: LayoutServerLoad = async ({ cookies, url }) => {
  const token   = cookies.get('phavo_session')
  const session = token ? await db.getSession(token) : null

  if (!session && url.pathname !== '/login' && !url.pathname.startsWith('/setup'))
    throw redirect(302, '/login')

  const config = await db.getConfig()
  if (session && !config.setupComplete && url.pathname !== '/setup')
    throw redirect(302, '/setup')

  // tier is intentionally NOT returned — upgrade prompts are cosmetic
  return { loggedIn: !!session, dashboardName: config.dashboardName }
}

Widget polling pattern

Widgets with refreshInterval > 0 are polled on a staggered schedule: each widget starts 1500ms after the previous one. This spreads CPU load on constrained hardware (Pi 3/4). Polling uses separate timeouts[] and intervals[] arrays so teardown can use clearTimeout for pending starts and clearInterval for running polls.

// widgets.svelte.ts — simplified
// polls start staggered (1500ms × index) to spread CPU load
const timeouts: ReturnType<typeof setTimeout>[] = []
const intervals: ReturnType<typeof setInterval>[] = []

activeDefinitions.forEach((def, i) => {
  const t = setTimeout(() => {
    intervals.push(setInterval(() => fetchWidgetData(def, true), def.refreshInterval))
  }, i * 1500)
  timeouts.push(t)
})

// teardown — clear pending starts separately from running polls
return () => {
  for (const id of timeouts) clearTimeout(id)
  for (const id of intervals) clearInterval(id)
}

Stale detection: if Date.now() − lastUpdated > 2 × refreshInterval on a failed poll, status becomes stale; otherwise error.

License Activation (Celestial)

Key format

// base64url( ed25519_signature(64 bytes) || utf8(json_payload) )

// Payload:
{
  "tier":      "celestial",
  "licenseId": "<uuid>",
  "issuedAt":  "<iso8601>"

Activation flow

POST /api/v1/license/activate  { key: string }

1. base64urlDecode(key) → signature(64 bytes) + payload bytes
2. ed25519Verify(signature, payloadBytes, embeddedPublicKey)  → 400 on fail
3. JSON.parse(payloadBytes) → { tier, licenseId, issuedAt }
4. INSERT licenseActivation { licenseKey: key, tier, licenseId, issuedAt, activatedAt: now() }
5. UPDATE sessions SET tier = 'celestial' WHERE id = currentSessionId
6. return { ok: true, data: { tier: 'celestial', licenseId, issuedAt } }

// On every server start:
// SELECT licenseActivation. If present: re-verify Ed25519. Fail → fall back to stellar.

Security Decisions Summary

#DecisionRationale
1requireTier() re-reads tier from DB on every request via licenseActivation lookupCookie modification has zero effect — tier is server-controlled
2Session token is 32-byte random base64url — no JWT, no payloadNothing to decode, nothing to forge
3No tier column in users or config tablesDB editing cannot elevate tier
4License key is Ed25519-signed payload — verified locally against two embedded public keys (current + next) on activation and on every server startFully offline. Tampered key fails signature verification → stellar behaviour.
5Widget manifest returns teasers (no dataEndpoint) for locked Celestial widgets when session is stellarClient has no information to call locked endpoints even with devtools
6Tier never serialised into client HTML or Svelte storesUpgrade prompts are cosmetic UX only — server enforces access independently
7Fonts self-hosted via @fontsource — CSP allowlists no external font CDNEliminates Google Fonts network dependency and privacy concern
8Plugin import allowlist enforced by static analysis at load timeMalicious plugins rejected before any code executes
9Plugin permissions declared in manifest.json — undeclared capabilities blocked at runtimeDefense-in-depth alongside import allowlist
10No telemetry. No analytics. No outbound calls at runtime except user-initiated actions.Privacy by design — zero data leaves the device unless user explicitly triggers it
11SSRF guard on all user-provided URLs — private IP ranges, localhost, and cloud metadata endpoints blockedPrevents widgets from being used to attack internal network
12execFile() used for all subprocesses — never exec()Prevents shell injection in update and Docker socket commands
